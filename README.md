# 圆小付系统接口测试项目

这是一个支付类系统的接口测试练习项目，主要围绕登录、转账、个人信息修改、基金购买这几条常见链路做测试设计和自动化回归。

项目里包含一个本地 mock 服务，方便在没有真实后端环境时复现接口测试流程。测试代码重点放在参数校验、鉴权、金额边界、重复提交、余额一致性和响应结构校验上。

## 项目内容

- `api/roundpay_server.py`：本地 mock 接口服务
- `tests/`：pytest 接口自动化测试用例
- `docs/`：测试计划、测试用例和缺陷记录示例
- `postman/`：Postman 集合
- `jmeter/`：基金购买链路性能测试脚本示例

## 覆盖范围

| 模块 | 测试重点 |
| --- | --- |
| 登录 | 正常登录、密码错误、必填参数、Token 返回 |
| 转账 | 金额边界、余额变化、重复提交、未登录访问 |
| 个人信息 | 手机号格式、昵称修改、鉴权校验 |
| 基金购买 | 金额下限、余额不足、基金状态、响应结构 |

## 技术栈

- Python 3.10+

- requests
- jsonschema
- Allure
- Postman

## 本地运行

安装依赖：

```bash
pip install -r requirements.txt
```

执行自动化测试：

```bash
pytest
```

生成 Allure 原始结果：

```bash
pytest --alluredir=reports/allure-results
```

如果本地已安装 Allure CLI，可以查看报告：

```bash
allure serve reports/allure-results
```

## 测试设计说明

这个项目没有追求复杂架构，主要是为了把支付类系统常见的测试点整理成可运行的用例。用例设计时重点考虑了以下问题：

- 转账金额为 0、负数、小数时是否能正确拦截
- 转账成功后付款方和收款方余额是否同时更新
- 同一个请求号重复提交时是否会重复扣款
- 未登录或 Token 错误时是否能拒绝访问
- 基金购买金额低于起购金额时是否有明确提示
- 接口返回字段是否稳定，避免前端解析异常

## 项目复盘

做这个项目时，我把测试重点放在支付类系统比较容易出问题的地方，例如金额边界、重复提交、鉴权失败和余额一致性。自动化用例没有只验证状态码，也补充了响应字段和业务结果校验，这样回归时更容易发现接口结构变化或业务逻辑异常。

后续如果继续完善，会补充数据库落库校验、更多异常交易场景，以及 Jenkins 定时执行自动化测试。
